# Dockerized Scoutpost Installer

This is the recommended self-host setup path for newsrooms that want a
disposable operator container instead of installing Node, Deno, Supabase CLI,
GitHub CLI, jq, and OpenSSL on the host.

For the full newsroom-facing install guide, including the tech stack, credentials
flow, validation, updates, and `scout` CLI setup, see
[`docs/oss/newsroom-docker-install.md`](../../docs/oss/newsroom-docker-install.md).

The container does not define a separate setup flow. It mounts the same
`scoutpost-setup.json` generated by `/setup` and runs
`selfhost/setup-from-manifest.sh` from the repository.

The same container is also the downstream maintenance operator: use it for
`doctor` before and after changes, and for `update` when preparing upstream
maintenance PRs in newsroom forks.

The `/setup` page downloads the easy path:

```bash
bash scoutpost-docker-install.sh install
```

Keep `scoutpost-docker-install.sh` beside `scoutpost-setup.json`. The
script pulls `ghcr.io/buriedsignals/scoutpost-installer:latest` when it can;
if the package is not readable yet, it clones `scoutpost-os` into the
workspace and builds the same installer image locally.

## Install

Run from the directory that contains `scoutpost-setup.json`. If `/workspace`
is not already a Scoutpost checkout, the image clones
`buriedsignals/scoutpost-os` into `/workspace/scoutpost-os`.

```bash
docker run --rm -it \
  -v "$PWD:/workspace" \
  -v "$PWD/scoutpost-setup.json:/config/scoutpost-setup.json:ro" \
  ghcr.io/buriedsignals/scoutpost-installer:latest install
```

For a local development build instead of the prebuilt image:

```bash
docker build -f deploy/installer/Dockerfile -t scoutpost-installer .
docker run --rm -it \
  -v "$PWD:/workspace" \
  -v "$PWD/scoutpost-setup.json:/config/scoutpost-setup.json:ro" \
  scoutpost-installer install
```

## Doctor

```bash
docker run --rm -it \
  -v "$PWD:/workspace" \
  -v "$PWD/scoutpost-setup.json:/config/scoutpost-setup.json:ro" \
  ghcr.io/buriedsignals/scoutpost-installer:latest doctor
```

## Update

Run from the downstream newsroom fork checkout. Mounting an empty deployment
directory is fine for initial install, but update PRs need a fork checkout with
a pushable `origin`.

```bash
docker run --rm -it \
  -v "$PWD:/workspace" \
  -v "$PWD/scoutpost-setup.json:/config/scoutpost-setup.json:ro" \
  -v "$HOME/.config/gh:/root/.config/gh:ro" \
  ghcr.io/buriedsignals/scoutpost-installer:latest update
```

`update` fetches upstream `master`, prepares a maintenance branch, refreshes
the weekly sync workflow, and opens a PR when GitHub CLI auth is mounted.
Secrets stay out of Git. The manifest remains local and mounted read-only.
Generated frontend secrets are written to `frontend/.env.production.local` so
newsroom forks can pull upstream changes without local config dirtying tracked
frontend files.
The installer does not run interactive Firecrawl browser authentication; the
manifest API key is the deployment credential. Optional local/provider CLIs are
only installed when `COJOURNALIST_INSTALL_AGENT_TOOLING=true` is set.
For Supabase Cloud installs, include `supabase.access_token` in
`scoutpost-setup.json`; the container exports it as `SUPABASE_ACCESS_TOKEN`
so Supabase CLI project creation, migration push, and Edge Function deploys can
run without browser login.
